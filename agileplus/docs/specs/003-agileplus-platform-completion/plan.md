# Implementation Plan: AgilePlus Platform Completion

**Branch**: `main` | **Date**: 2026-03-02 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `kitty-specs/003-agileplus-platform-completion/spec.md`

## Summary

End-to-end platform completion for AgilePlus: bidirectional Plane.so sync engine, platform service infrastructure (NATS, Dragonfly, Neo4j, MinIO via Process Compose), event-sourced persistence with hash-chained audit trail, htmx+Alpine.js web dashboard served from Rust axum, CLI integration, and multi-device sync via Tailscale P2P + git-backed state.

Approach: extend the existing hexagonal Rust workspace (11 crates) with 7 new crates and enhance 4 existing ones. Server-rendered dashboard (no frontend build pipeline). Process Compose for orchestration. SQLite+WAL as event store with NATS JetStream for inter-service messaging.

## Technical Context

**Language/Version**: Rust nightly (2024 edition), Python 3.14+ (MCP), TypeScript 7.0+ (optional tooling)
**Primary Dependencies**: axum 0.7, async-nats 0.35, neo4rs 0.7, redis 0.25 (Dragonfly), aws-sdk-s3 1.15, sqlx 0.8, askama 0.12, axum-htmx 0.8, opentelemetry-otlp 0.31, tailscale-localapi 0.1
**Storage**: SQLite (WAL mode) for events/state, Dragonfly for cache, Neo4j for graph, MinIO for artifacts
**Testing**: cargo test + cucumber-rs (BDD) + pact-rust (contract) + proptest (property-based)
**Target Platform**: macOS (primary), Linux, Windows
**Project Type**: Rust workspace with multiple crates
**Performance Goals**: <50ms CLI startup, <5ms SQLite queries, <100ms API p95, <10ms gRPC localhost
**Constraints**: <100MB idle memory, local-first (no cloud deps), Process Compose (not Docker)
**Scale/Scope**: 100K events, 500 features per project, 10+ parallel agents

### Data Flow

```
CLI command → Domain validation → Event emitted → NATS JetStream
  ├── Event Store (SQLite) ← event persisted
  ├── Cache (Dragonfly) ← projection updated
  ├── Graph (Neo4j) ← relationships updated
  ├── Dashboard (SSE) ← real-time push to browser
  └── Sync Engine → Plane.so API (outbound)

Plane.so webhook → Webhook endpoint → Sync Engine
  ├── Conflict check (content hash comparison)
  ├── Event emitted → same flow as above
  └── ACK to Plane.so
```

## Work Package Grouping (high-level)

These groups guide `/spec-kitty.tasks` decomposition:

### Group 1: Foundation (P1, sequential)

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Neo4j JVM memory (500MB+) | High memory on dev machines | Process Compose memory limits; graph is optional (degrade to SQLite-only) |
| Plane.so rate limit (60/min) | Throttled bulk sync | Batch API calls, exponential backoff, sync queue with rate limiter |
| NATS JetStream file corruption | Lost events | Snapshots as recovery points; MinIO archive for old events |
| htmx+Alpine.js complexity boundary | Unclear responsibilities | Convention doc: Alpine for client state, htmx for server communication |
| Cross-device SQLite WAL | WAL doesn't work over network FS | P2P replicates events (not DB file); git serializes to JSON |
