## R1: Task Runner Evaluation

### Decision: Evaluate mise as primary candidate; moon as alternative

### Rationale

Scored candidates against org requirements (47 repos, Rust/Python/TypeScript/Go, standardized targets, language auto-detection, CI+local, plugin architecture):

| Tool | Standardized Targets | Auto-detection | CI/Local | Plugins | Monorepo | Overall |
|------|---------------------|----------------|----------|---------|----------|---------|
| **mise** | 8/10 | 6/10 | 9/10 | 10/10 | 8/10 | **8.2/10** |
| **moon** | 9/10 | 9/10 | 9/10 | 2/10 | 10/10 | **7.8/10** |
| **Taskfile** | 7/10 | 3/10 | 9/10 | 1/10 | 4/10 | **4.8/10** |
| **just** | 6/10 | 2/10 | 8/10 | 1/10 | 2/10 | **3.8/10** |
| **xc** | 5/10 | 1/10 | 5/10 | 1/10 | 2/10 | **2.8/10** |
| **earthly** | — | — | — | — | — | **Unmaintained** |
| **teax** | — | — | — | — | — | **Not found** |

**mise** wins on plugin architecture (1000+ asdf plugins) and polyglot tool version management. New monorepo tasks feature (Oct 2025, experimental) enables unified task discovery across repos. TOML config familiar to Rust/Python devs.

**moon** is strongest for dependency graphs and caching but has no plugin system and is web-ecosystem biased. Strong alternative if mise monorepo tasks don't stabilize.

**teax** — no public tool found under this name as of March 2026. May be private/unreleased.

### Alternatives Rejected

- **Taskfile**: Single-repo focus, no language detection, no plugins, no monorepo orchestration
- **just**: No monorepo, no polyglot awareness, no plugin system
- **xc**: Too small ecosystem, no dependency graph, no CI integration
- **earthly**: Unmaintained as of 2026

## R2: Pre-Release Version Conventions by Registry

### Decision: Use registry-native conventions for each tier

### Channel → Version Mapping

| Registry | alpha | canary | beta | rc | prod |
|----------|-------|--------|------|----|------|
| **npm** | `X.Y.Z-alpha.N` `--tag alpha` | `X.Y.Z-canary.N` `--tag canary` | `X.Y.Z-beta.N` `--tag beta` | `X.Y.Z-rc.N` `--tag rc` | `X.Y.Z` `--tag latest` |
| **PyPI** | `X.Y.ZaN` | `X.Y.Z.devN` | `X.Y.ZbN` | `X.Y.ZrcN` | `X.Y.Z` |
| **crates.io** | `X.Y.Z-alpha.N` | `X.Y.Z-canary.N` | `X.Y.Z-beta.N` | `X.Y.Z-rc.N` | `X.Y.Z` |
| **Go proxy** | `vX.Y.Z-alpha.N` | `vX.Y.Z-canary.N` | `vX.Y.Z-beta.N` | `vX.Y.Z-rc.N` | `vX.Y.Z` |
| **Hex.pm** | `X.Y.Z-alpha.N` | `X.Y.Z-canary.N` | `X.Y.Z-beta.N` | `X.Y.Z-rc.N` | `X.Y.Z` |
| **Zig** | Git tags (no registry) | Git tags | Git tags | Git tags | Git tags |
| **Mojo** | N/A (no registry) | N/A | N/A | N/A | Manual |

### Key Notes

- **PyPI canary → `devN`**: PEP 440 has no "canary" concept. `devN` sorts before `aN` which matches canary's position (before alpha stability-wise). Alternative: use `aN` for both alpha and canary with different N ranges.
- **npm dist-tags**: Separate from version — npm uniquely supports both a version string AND a named tag. `npm install pkg@canary` uses the dist-tag.
- **All registries**: Pre-releases require explicit opt-in by consumers. No registry auto-upgrades to pre-releases.
- **Zig/Mojo**: No centralized registries. Pre-wire adapters should use git-tag-based versioning.

## R3: Phenotype CLI Technology Stack

### Decision: Go with Cobra + Lipgloss/Bubbles + Viper

### Rationale

- **Cobra**: Industry standard for Go CLIs (kubectl, docker, gh). Nested subcommands, flag inheritance, auto-generated help.
- **Viper**: Native Cobra integration, reads TOML/YAML/JSON config files.
- **Lipgloss + Bubbles** (Charm ecosystem): Styled tables, progress bars, colored output without full TUI overhead.
- **Single binary**: No runtime dependencies, fast startup, cross-platform.

### Alternatives Rejected

- **urfave/cli**: Simpler but lacks sophistication for complex multi-tier commands
- **Kong**: Smaller ecosystem, struct-tag approach less conventional
- **Bubble Tea**: Full TUI framework — overkill for a standard CLI

### CLI Structure

```
pheno
├── publish      # Publish package to registry
├── promote      # Promote package to next channel
├── audit        # Org-wide release status report
├── bootstrap    # Initialize governance artifacts in a repo
├── matrix       # Generate release matrix
└── config       # Manage CLI configuration
```

## R4: CI Workflow Architecture

### Decision: Hybrid — centralized reusable workflows + per-repo wrappers

### Rationale

- **Centralized** (in `phenotypeActions` repo): Core logic for publishing, gate checks, promotion. Single source of truth. Version-pinned via tags (`@v1`, `@v2`).
- **Per-repo wrappers**: Generated by `pheno bootstrap`. Call into centralized workflows with repo-specific inputs (language, registry, risk profile). Can override individual steps.

### Workflow Catalog (centralized)

| Workflow | Trigger | Purpose |
|----------|---------|---------|
| `publish.yml` | Called by per-repo | Registry-specific publishing with retry/backoff |
| `gate-check.yml` | Called by per-repo | Run channel-specific gate criteria |
| `promote.yml` | Tag push or dispatch | Orchestrate gate-check → publish |
| `changelog.yml` | Release tag | Generate changelog via git-cliff |
| `audit.yml` | Scheduled/dispatch | Org-wide release status report |

## R5: Plugin/Adapter Architecture for Registry Support

### Decision: Registry adapter interface with per-registry implementations

### Rationale

The Phenotype CLI needs to support 4 active registries now (npm, PyPI, crates.io, Go proxy) and pre-wire 3 future ones (Hex.pm, Zig, Mojo). A plugin/adapter pattern allows:

1. Each registry has a consistent interface: `detect()`, `version()`, `build()`, `publish()`, `verify()`
2. Language detection maps to registry adapters
3. New registries added by implementing the adapter interface
4. Pre-wired adapters for future registries return "not yet supported" gracefully

### Adapter Manifest Detection

| Language | Manifest | Registry | Adapter Status |
|----------|----------|----------|---------------|
| Rust | `Cargo.toml` | crates.io | Active |
| Python | `pyproject.toml` | PyPI | Active |
| TypeScript/Node | `package.json` | npm | Active |
| Go | `go.mod` | Go proxy | Active |
| Elixir | `mix.exs` | Hex.pm | Pre-wired |
| Zig | `build.zig.zon` | Git tags | Pre-wired |
| Mojo | `mojoproject.toml` | N/A | Pre-wired (stub) |
